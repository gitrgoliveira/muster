## Clarifications

### Session 2026-05-22

- Q: What format should generated bead IDs use? → A: `bd-` + first 4 hex chars of a UUIDv4, matching the beads system naming convention (e.g., `bd-a1f2`).
- Q: What is the Go module name? → A: `github.com/gitrgoliveira/muster` — the project owner's GitHub org is `gitrgoliveira`.
- Q: Does `/move` enforce column transition constraints? → A: Unrestricted in M0 — any valid column name accepted; lifecycle gating deferred to M1+.
- Q: Should M0 WS hub implement `seq` counter + `?since=` replay buffer? → A: No — deferred to M1. M0 hub broadcasts events only; disconnected clients re-fetch via `GET /api/v1/beads`.
- Q: What logging approach for `muster serve`? → A: `log/slog` text format to stderr; startup banner (port, build, schemaVersion) printed to stdout.

### Session 2026-05-22 (round 2 — conflict reconciliation)

- Q: Should PATCH with an empty body return 200 (no-op) or 400 INVALID_REQUEST? → A: `400 INVALID_REQUEST` (consistent with §Edge Cases) — at least one field must be present.
- Q: Should `/move` support `beforeID` positional insertion in M0? → A: Yes — implemented in M0. Body `{"toColumn", "beforeID?"}`; absent `beforeID` appends to end of `toColumn`. See US6 acceptance scenarios 3–5.
- Q: Should the in-memory store also hold providers + capacity + dolt seed datasets in M0? → A: Yes — all four datasets (beads, providers, capacity, dolt) populated at startup. Providers/capacity endpoints come in M1; `/orchestrator/status` returns the seeded dolt object in M0.
- Q: How are coverage targets reconciled between this spec and plan.md? → A: `plan.md §Test Coverage Targets` is authoritative. SC-006 references the plan; SC-007 changed to "race detector passes".

### Session 2026-05-22 (round 3 — P1 gap closure)

User-decided (4 items):

- Q: Seed timestamp format? → A: **Opaque strings** — `at`/`createdAt`/etc. are documented as opaque; no parsing required. Seed retains prototype's calendar strings (`Mon 09:14`); new events generated by handlers use RFC3339. UI treats them as labels.
- Q: Request body size limit? → A: **1 MiB cap** via `http.MaxBytesReader` wrapped around POST/PATCH handlers. Exceeded → **400 `INVALID_REQUEST`** with message `request body exceeds 1 MiB limit` (consistent with the existing error-code matrix; no new HTTP status or code added).
- Q: Coverage targets — gate or aspiration? → A: **Gates** — CI fails below the per-package thresholds in `plan.md §Test Coverage Targets`.
- Q: Lint/format stack? → A: **`gofmt + go vet + golangci-lint` with a pinned, curated config** enabling: errcheck, govet, gofmt, gosimple, ineffassign, staticcheck, unused. Config lives at `.golangci.yml`.

Default-resolved (the rest of the P1 list — sensible defaults applied without further input):

- Graceful shutdown → on `SIGINT`/`SIGTERM`, call `http.Server.Shutdown(ctx)` with a 5-second drain timeout. WS connections receive a normal close frame (status code 1001 — "going away"). Process exits 0 after shutdown completes; exit 1 if drain timeout elapses.
- `--addr` flag → parsed via `net.SplitHostPort`. IPv6 addresses supported (`[::1]:7766`). Invalid format → log error to stderr and exit 1. Port already in use → log `bind: address already in use` and exit 1.
- Title length → 255 **runes** (via `utf8.RuneCountInString`), not bytes. Allows the same character budget regardless of UTF-8 multi-byte content.
- Validation order → structural (malformed JSON → unknown field → null where forbidden) → required-field presence → enum membership → numeric range. First failure short-circuits and is returned.
- WS max frame size → 1 MiB (matches request body cap; configured via `websocket.AcceptOptions{InsecureSkipVerify: false, OriginPatterns: nil, CompressionMode: websocket.CompressionDisabled}` and a per-connection read limit).
- Enums for free-form prototype fields:
  - `NowPlaying.Kind` → `"tool" | "thought" | "output"` (typed as `core.NowPlayingKind`).
  - `LogEntry.Kind` → `"system" | "tool" | "thought" | "output"` (typed as `core.LogKind`).
  - `FileChange.Status` → `"A" | "M" | "D"` (typed as `core.FileStatus`).
- SC-002 latency budget → **p95 ≤ 50 ms** over 100 sequential requests on a warm server (after at least 5 warm-up requests).
- SC-005 latency budget → **p95 ≤ 100 ms** measured from the REST 2xx response write to the WS frame received by a connected client. Sample of 50 mutations.
- FR-001 acceptance → `file ./muster` reports a Mach-O / ELF binary; running `./muster --version` (or `--help`) emits the build banner. Embedded UI is served at `/` after the binary is moved to a different directory (proves self-containment).
- FR-015 acceptance → `curl -sf http://localhost:7766/api/v1/healthz | jq '.ok'` returns `true`.
- FR-017 acceptance → for every endpoint defined in `contracts/rest-api.md`, `curl -i ... | grep -i 'x-request-id:'` returns one header. When supplied on request, the value is echoed; when absent, a fresh UUIDv4 is generated.
- FR-020 acceptance → `go.mod` lists the four pinned dependencies at the versions in `research.md §Decision 9`.
- Pre-merge gate → all four must pass in CI: (a) `go build ./...` clean, (b) `go test -race ./...` green, (c) `go tool cover -func=cover.out` meets per-package thresholds, (d) `gofmt -l .` empty + `go vet ./...` clean + `golangci-lint run` clean.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Start muster and view the UI (Priority: P1)

A developer clones the repo, runs `go run ./cmd/muster serve`, and opens `http://localhost:7766` in their browser. The embedded prototype UI (Muster.html + styles.css + JSX files) loads and renders the Kanban board with seed data pre-populated.

**Why this priority**: Without serving the UI, nothing else matters. This proves the binary works.

**Independent Test**: Run `muster serve`, open `http://localhost:7766/`, confirm the page loads with the prototype HTML and CSS.

**Acceptance Scenarios**:

1. **Given** the binary is built, **When** the user runs `muster serve`, **Then** the server binds to `127.0.0.1:7766` and logs a startup message.
2. **Given** the server is running, **When** the user opens `http://localhost:7766/`, **Then** `Muster.html` is served along with all its static assets (`styles.css`, `*.jsx`, font imports).
3. **Given** the server is running, **When** the user opens `http://localhost:7766/` in a browser, **Then** the prototype renders identically to serving the files via `python3 -m http.server`.

---

### User Story 2 - List beads via REST API (Priority: P1)

The UI (or `curl`) calls `GET /api/v1/beads` and receives a JSON array of all seed beads matching the `Bead` shape from `spec.md §3.1` and `DESIGN.md §1.11`.

**Why this priority**: The REST API is the primary contract between UI and backend — the board can't render without bead data.

**Independent Test**: `curl http://localhost:7766/api/v1/beads | jq '.items | length'` returns `14` (the number of TASKS in seed data).

**Acceptance Scenarios**:

1. **Given** the server is running with seed data, **When** `GET /api/v1/beads` is called, **Then** the response is `200 OK` with `Content-Type: application/json` and body `{"items": [...], "nextCursor": null, "total": 14}`.
2. **Given** the response, **When** examining any bead, **Then** it carries all fields from the `Bead` type: `id`, `title`, `desc`, `type`, `column`, `priority`, `labels`, `vcs`, `steps`, `history`, `tokensUsed`, `tokensBudget`, `createdAt`, `lastActivity`, `ready`, `repo`, `estimate`, `assignee`, `acceptance`, `comments`.
3. **Given** the server is running, **When** `GET /api/v1/beads?column=running` is called, **Then** only beads in the `running` column are returned.

---

### User Story 3 - Get a single bead by ID (Priority: P2)

`GET /api/v1/beads/{id}` returns the full bead object for one bead, including `log`, `files`, and `diffPreview` (seed data).

**Why this priority**: The drawer view needs per-bead detail; a separate endpoint allows lazy loading.

**Independent Test**: `curl http://localhost:7766/api/v1/beads/bd-a1f2 | jq '.id'` returns `"bd-a1f2"`.

**Acceptance Scenarios**:

1. **Given** `bd-a1f2` exists, **When** `GET /api/v1/beads/bd-a1f2` is called, **Then** the response is `200 OK` with the full bead.
2. **Given** the ID `bd-nonexistent`, **When** `GET /api/v1/beads/bd-nonexistent` is called, **Then** the response is `404` with `{"error": {"code": "BEAD_NOT_FOUND", ...}}`.

---

### User Story 4 - Create a new bead (Priority: P2)

`POST /api/v1/beads` with a JSON body creates a new bead in the in-memory store and returns it with a generated ID.

**Why this priority**: The "New Bead" composer in the UI needs the create endpoint.

**Independent Test**: POST a new bead, then GET `/api/v1/beads` and confirm the count increased by 1.

**Acceptance Scenarios**:

1. **Given** a valid CreateBeadRequest body, **When** `POST /api/v1/beads` is called, **Then** the response is `201 Created` with the full bead including a generated ID of the form `bd-XXXX` (first 4 hex chars of a UUIDv4, matching the beads system's naming convention).
2. **Given** a body missing `title`, **When** `POST /api/v1/beads` is called, **Then** the response is `400 INVALID_REQUEST`.

---

### User Story 5 - Update (PATCH) a bead (Priority: P2)

`PATCH /api/v1/beads/{id}` accepts a sparse JSON body and updates only the supplied fields.

**Why this priority**: The drawer's inline-edit features (title, labels, priority, etc.) need partial update.

**Independent Test**: PATCH a bead's title, then GET it and confirm the title changed.

**Acceptance Scenarios**:

1. **Given** `bd-a1f2` exists, **When** `PATCH /api/v1/beads/bd-a1f2` with `{"title": "New title"}`, **Then** `200 OK` with the updated bead, title changed, other fields untouched.
2. **Given** a nonexistent ID, **When** PATCH is called, **Then** `404 BEAD_NOT_FOUND`.

---

### User Story 6 - Move a bead between columns (Priority: P2)

`POST /api/v1/beads/{id}/move` with `{toColumn, beforeID?}` moves a bead to a different column in the in-memory store and emits a WS event.

**Why this priority**: Drag-and-drop on the board fires this endpoint.

**Independent Test**: Move a bead from `backlog` to `scheduled`, then GET it and confirm the column changed.

**Acceptance Scenarios**:

1. **Given** `bd-b210` is in `backlog`, **When** `POST /api/v1/beads/bd-b210/move {"toColumn":"scheduled"}`, **Then** `200 OK` and the bead is now in `scheduled` (appended at the end of that column).
2. **Given** the move, **Then** a `bead.moved` WS event is emitted to all connected clients.
3. **Given** `bd-b210` is in `backlog` and `bd-4f12` is also in `backlog` ahead of `bd-2d55`, **When** `POST /api/v1/beads/bd-b210/move {"toColumn":"backlog","beforeID":"bd-2d55"}`, **Then** `200 OK` and `bd-b210` is reordered to appear immediately before `bd-2d55` within `backlog`.
4. **Given** an unknown `beforeID`, **When** the move is called, **Then** `400 INVALID_REQUEST` with message referencing the unknown ID.
5. **Given** `beforeID` references a bead in a different column than `toColumn`, **When** the move is called, **Then** `400 INVALID_REQUEST` (beforeID must be in the destination column).

---

### User Story 7 - WebSocket event stream (Priority: P1)

`GET /api/v1/stream` upgrades to WebSocket. The server sends a `hello` event within 1s, then pushes `bead.updated`, `bead.moved`, `bead.created`, `bead.deleted` events as mutations happen via REST.

**Why this priority**: The UI needs live updates; without WS the board goes stale.

**Independent Test**: Connect a WS client, create a bead via REST, confirm the WS client receives a `bead.created` event.

**Acceptance Scenarios**:

1. **Given** a WS connection, **Then** the server sends `{"type":"hello","build":"dev","schemaVersion":1,...}` within 1 second.
2. **Given** a WS connection, **When** a bead is created via REST, **Then** the WS receives `{"type":"bead.created","bead":{...}}`.
3. **Given** a WS connection, **When** a bead is moved via REST, **Then** the WS receives `{"type":"bead.moved","id":"...","fromColumn":"...","toColumn":"..."}`.
4. **Given** a WS client sends `{"type":"ping"}`, **Then** the server responds with `{"type":"pong","at":"..."}`.

---

### User Story 8 - Dispatch a bead (Priority: P3)

`POST /api/v1/beads/{id}/dispatch` transitions a bead from `scheduled` to `running` in the in-memory store, appends `claimed` + `started` history events, and emits WS events. (No real agent invocation in M0 — just state transitions.)

**Why this priority**: Enables testing the dispatch flow end-to-end in the UI even before real CLI adapters exist.

**Independent Test**: Dispatch a scheduled bead, confirm its column is `running` and history has `claimed`+`started` entries.

**Acceptance Scenarios**:

1. **Given** `bd-7c0d` is in `scheduled`, **When** `POST /api/v1/beads/bd-7c0d/dispatch`, **Then** `200 OK`, column is `running`, history includes `claimed` and `started` events.
2. **Given** a bead in `backlog`, **When** dispatch is called, **Then** `400 INVALID_STATE`.

---

### User Story 9 - Add a comment (Priority: P3)

`POST /api/v1/beads/{id}/comments` with `{"text":"..."}` appends a `comment` history event.

**Why this priority**: Drawer comment thread needs persistence.

**Independent Test**: Post a comment, GET the bead, confirm `history` includes a `comment` event and `comments` count incremented.

**Acceptance Scenarios**:

1. **Given** `bd-a1f2`, **When** `POST /api/v1/beads/bd-a1f2/comments {"actor":"you@yours.dev","note":"LGTM"}`, **Then** `201` with the updated bead (including the appended history event and incremented comment count), and a `comment.added` WS event is emitted with payload `{id, event, bead}`.

---

### User Story 10 - Orchestrator status & healthz (Priority: P3)

`GET /api/v1/orchestrator/status` returns daemon metadata. `GET /api/v1/healthz` returns 200.

**Why this priority**: UI needs status for the Dolt chip and for version checks.

**Independent Test**: `curl http://localhost:7766/api/v1/healthz` returns 200. `curl http://localhost:7766/api/v1/orchestrator/status | jq '.build'` returns a string.

**Acceptance Scenarios**:

1. **Given** the server is running, **When** `GET /api/v1/healthz`, **Then** `200 OK`.
2. **Given** the server is running, **When** `GET /api/v1/orchestrator/status`, **Then** `200 OK` with `{build, schemaVersion, beadsVersion, online, serverTime, dolt}`.

---

### Edge Cases

- What happens when the server can't bind to port 7766? (Already in use) — Log error and exit with non-zero.
- What happens when WS clients disconnect abruptly? — Hub removes them from broadcast list.
- What happens when multiple WS clients are connected? — All receive the same events.
- What happens when PATCH sends an empty body? — Return 400 INVALID_REQUEST.
- What happens when POST /move uses an invalid column name (not one of the 5 valid values)? — Return 400 INVALID_REQUEST. Any of the 5 valid columns (`backlog`, `scheduled`, `running`, `review`, `done`) are accepted regardless of current column.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST serve a single Go binary (`muster`) with the UI embedded via `go:embed`.
- **FR-002**: System MUST bind to `127.0.0.1:7766` by default (configurable via `--addr` flag).
- **FR-003**: System MUST serve the prototype files (`Muster.html`, `styles.css`, `*.jsx`) as static assets at the root path.
- **FR-004**: System MUST expose `GET /api/v1/beads` returning in-memory bead data matching the `Bead` shape from `spec.md §3.1`.
- **FR-005**: System MUST populate the in-memory store with the 14 seed beads from `prototype/data.jsx` TASKS array.
- **FR-006**: System MUST expose `GET /api/v1/beads/{id}` for single-bead detail.
- **FR-007**: System MUST expose `POST /api/v1/beads` for creating new beads. Generated IDs MUST use the beads naming format: `bd-` + first 4 hex chars of a UUIDv4 (e.g., `bd-a1f2`).
- **FR-008**: System MUST expose `PATCH /api/v1/beads/{id}` for partial updates.
- **FR-009**: System MUST expose `POST /api/v1/beads/{id}/move` for column transitions. The request body is `{"toColumn": "<col>", "beforeID": "<id>"}` where `toColumn` is required and `beforeID` is optional. Any of the 5 valid column names is accepted as `toColumn` — no lifecycle ordering is enforced for the column transition itself (deferred to M1). When `beforeID` is supplied it MUST reference an existing bead in `toColumn`; the moved bead is inserted immediately before it. When `beforeID` is absent, the moved bead is appended at the end of `toColumn`.
- **FR-010**: System MUST expose `POST /api/v1/beads/{id}/dispatch` for scheduling→running transitions (in-memory state only, no real agent).
- **FR-011**: System MUST expose `POST /api/v1/beads/{id}/comments` for comment creation.
- **FR-012**: System MUST expose `GET /api/v1/stream` as a WebSocket endpoint that pushes typed events (`hello`, `bead.created`, `bead.updated`, `bead.moved`, `bead.deleted`, `comment.added`, `pong`). The hub broadcasts only — no `seq` counter, no `?since=` replay buffer in M0 (deferred to M1).
- **FR-013**: System MUST send a `hello` event within 1 second of WS connection.
- **FR-014**: System MUST respond to `ping` frames with `pong` events.
- **FR-015**: System MUST expose `GET /api/v1/healthz` returning 200.
- **FR-016**: System MUST expose `GET /api/v1/orchestrator/status` returning daemon metadata.
- **FR-017**: All responses MUST include `X-Request-ID` headers.
- **FR-018**: Error responses MUST follow the shape `{"error": {"code": "...", "message": "...", "requestID": "..."}}`.
- **FR-019**: System MUST support `?column=` filter on `GET /api/v1/beads`.
- **FR-020**: System MUST use `go-chi/chi` for routing and `coder/websocket` for WS.

### Key Entities

- **Bead**: A unit of work with `id`, `title`, `type`, `column`, `priority`, `steps`, `history`, etc. (full shape in `spec.md §3.1` and `DESIGN.md §1.11`).
- **Step**: An ordered agent invocation within a bead's chain: `{agent, mode, skills[], status}`.
- **HistoryEvent**: Lifecycle events: `{at, kind, actor, agent?, note?}`.
- **Provider**: An LLM agent configuration (not CRUD'd in M0, just served as reference data).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `muster serve` starts and binds within 2 seconds.
- **SC-002**: `GET /api/v1/beads` returns 14 beads (matching seed data count) with <50ms latency.
- **SC-003**: The prototype UI loads at `http://localhost:7766/` and renders the board with seed data.
- **SC-004**: A WS client receives a `hello` event within 1 second of connection.
- **SC-005**: Creating a bead via REST produces a `bead.created` WS event within 100ms.
- **SC-006**: Unit test coverage meets the per-package targets defined in `plan.md §Test Coverage Targets` (core ≥80%, services ≥80%, store ≥80%, ws ≥75%, render/middleware ≥90%, api handlers ≥70%). The plan is the authoritative source; any change to a target updates the plan and is reflected here by reference.
- **SC-007**: `go test -race ./...` passes — no data races detected by the race detector.
- **SC-008**: Integration tests pass: full lifecycle (create → move → dispatch → comment → verify WS events).

## Assumptions

- Go 1.26+ is installed on the developer's machine.
- Logging uses `log/slog` with text handler to stderr. Startup prints a human-readable banner to stdout: `muster listening on http://127.0.0.1:7766 (build=dev schemaVersion=1)`. JSON format and configurable log level are deferred to M1.
- No external database is needed for M0 — all data is in-memory and lost on restart.
- The UI prototype files are embedded as-is (Babel-in-browser); no build step for the UI in M0.
- No authentication or CORS restrictions for M0 (localhost only).
- No real agent dispatching — `dispatch` just transitions state.
- No Dolt/Beads integration in M0 — that's M1.
- No tmux integration in M0 — that's M2.

## Technical Context

### Module Layout

```
cmd/muster/             main.go, --addr flag, go:embed ui/*

internal/
  core/                  pure domain types — Bead, Step, SubBead, HistoryEvent, enums, ID gen, validation
  store/                 in-memory store (slice-backed, sync.RWMutex), seed data
  services/              business logic — only layer that mutates store state, publishes WS events
  ws/                    WebSocket hub — Run loop, Client pumps, broadcast
  api/                   HTTP transport layer:
    render/              JSON + error response helpers
    middleware/          X-Request-ID middleware
    beads/               /api/v1/beads handlers + DTOs
    stream/              /api/v1/stream WS upgrade handler
    health/              /api/v1/healthz, /api/v1/orchestrator/status

ui/                      go:embed target — copy of prototype/ files
```

See `plan.md §Source Code Layout` for the authoritative file-by-file breakdown.

### Go Module

Module path: `github.com/gitrgoliveira/muster` (matching the gitrgoliveira GitHub org that owns the beads ecosystem).

### Key Dependencies

- `github.com/go-chi/chi/v5` — routing
- `github.com/coder/websocket` — WebSocket
- `github.com/google/uuid` — request IDs
- `github.com/stretchr/testify` — test assertions

### Seed Data

The in-memory store MUST be populated on startup with the exact data from `prototype/data.jsx`:
- 14 beads (TASKS array) with all derived fields (estimate, assignee, history, acceptance, comments, repo)
- 4 providers (AGENTS array) — served via a future `/providers` endpoint but included in the store
- 4 capacity entries (CAPACITY array) — served via a future `/orchestrator/capacity` endpoint
- The DOLT status object — served via `/orchestrator/status`

### API Versioning

All endpoints are under `/api/v1/`. JSON request/response, UTF-8, RFC3339 timestamps. The pagination envelope is `{"items": [...], "nextCursor": null, "total": N}`.

### Error Codes (M0 subset)

| Code | HTTP | When |
|------|------|------|
| `BEAD_NOT_FOUND` | 404 | ID doesn't match |
| `NOT_FOUND` | 404 | Unknown `/api/v1/*` path (non-bead) |
| `INVALID_STATE` | 400 | Lifecycle precondition not met (e.g., dispatch from non-`scheduled` column) |
| `INVALID_REQUEST` | 400 | Body validation failed, malformed JSON, unknown field, null where forbidden, invalid enum, empty PATCH body |
| `METHOD_NOT_ALLOWED` | 405 | Wrong HTTP verb on a known path |
| `INTERNAL` | 500 | Unexpected error (panic, ID generation exhausted after 3 retries) |
